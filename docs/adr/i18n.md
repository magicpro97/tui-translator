# I18N-01 — Cross-platform i18n architecture

- Status: Accepted
- Date: 2026-05-26
- Issue: [#481](https://github.com/magicpro97/tui-translator/issues/481)
- Roadmap ledger: `.github/steps/linux-cross-platform-quality-roadmap.md`

## Context

The tui-translator TUI ships with English strings hard-coded in
`src/tui/mod.rs`.  The cross-platform quality roadmap (parent epic
**HC-01**, **STD-02**) calls for a real localisation layer so that:

1. Vietnamese and future locales can replace user-facing English text
   without code changes.
2. Locale changes apply live via the existing `R` reload key and the
   `notify`-based config watcher (no TUI restart).
3. Adaptive-layout work (UX-01) is exercised by a pseudo-locale that
   inflates strings and exposes truncation.
4. Missing translations are surfaced loudly — CI hard-fails on missing
   keys; runtime falls back to English with a tracing warning rather
   than silently returning an empty string.

A small `t("key")` API is enough; we do not need full ICU MessageFormat
or runtime catalog discovery for V1.

## Decision

Adopt **Project Fluent** (`fluent-bundle` 0.15 + `unic-langid` 0.9) as
the i18n runtime.  Catalogs ship as `.ftl` files under `locales/`,
embedded into the binary via `include_str!` so the released `.exe`
needs no extra assets.

The new `src/i18n/` module exposes:

| API | Purpose |
|---|---|
| `i18n::init()` | Build the global catalog before the first frame. |
| `i18n::set_locale(tag)` | Switch the active locale (called from `apply_runtime_config`). |
| `i18n::t(key)` | Lookup with explicit fallback to en-US + tracing warning. |
| `i18n::t_arg(key, name, value)` / `t_args(key, args)` | Lookup with Fluent arguments. |
| `Locale` enum | Closed set `{EnUS, ViVN, Pseudo}` — adding a locale is a deliberate code change. |

### Why Fluent rather than the alternatives

- **`rust-i18n`** — pure compile-time macro over JSON/YAML; ergonomic
  but no proper plural/gender support and no pseudo-locale primitive.
  Rejected because the roadmap explicitly calls for pseudo-locale.
- **`gettext-rs`** — requires `libgettext` at runtime, which is a no-go
  for the single-`.exe` distribution model.
- **`fluent-rs` (fluent-bundle)** — pure Rust, supports pluralisation,
  interpolation, concurrent bundles, and lets us embed catalogs.  This
  is what we picked.

### Locale = config field, not a CLI flag

`AppConfig::locale` is a hot-reloadable field validated by
`AppConfig::validate` against the closed set `{en-US, vi-VN, x-pseudo}`.
Unknown tags are rejected by the validator, and any tag accepted by the
validator that the i18n loader doesn't know about still falls back to
`en-US` with a tracing warning so we never crash the TUI on a
config-typo.

### Live switching

`apply_runtime_config` already runs on:

- startup, with the persisted config;
- the `R` user action;
- file-system watcher events from `notify`;
- the onboarding-editor save path.

Adding a single `crate::i18n::set_locale(&effective_cfg.locale)` call to
that function means every existing apply path now drives the catalog.
No new event channel is required.

### Migration allowlist (I18N-01 scope)

I18N-01 ships the **architecture** and a minimal slice of migrated
strings.  The full TUI migration belongs to follow-up issues (STD-02
and HC-01 successor tickets) once the wave-6 sequencing window opens.

Migrated in this PR:

- `src/tui/mod.rs::render_help_overlay` — every visible row plus the
  title bar (scrollable and static variants).

Explicitly **not** migrated (documented allowlist):

- `StatusMetricsStrip` compact/expanded format strings.  Existing tests
  (`status_bar_*`, expanded-metrics suite) assert on exact substrings
  like `Lang:`, `Route:vmic:…`, and `missing virtual mic`.  Migrating
  these would require updating dozens of tightly coupled assertions
  outside the I18N-01 scope.  The strings remain English-only until a
  follow-up ticket coordinates the test churn.
- `render_language_prompt`, settings editor, onboarding screen, error
  toasts.
- All log messages (`tracing::info!`, `tracing::warn!`).  Operator logs
  stay in English for SRE-friendly grepping.

### Missing-key gate

`scripts/i18n-check.ps1` runs in CI (`.github/workflows/i18n-check.yml`)
and locally.  It enforces both directions:

1. Every `crate::i18n::t("…")` / `t_arg("…", …)` reference must exist
   in `locales/en-US.ftl`.
2. Every key in `locales/en-US.ftl` must exist in every other shipped
   catalog (currently only `vi-VN`).

The runtime fallback is still in place as defence-in-depth: even if a
release somehow shipped with a missing key, the user sees the English
string plus a tracing warning rather than an empty cell.

## Consequences

- **Positive**: Vietnamese users get a localised help overlay
  immediately, and the i18n layer is in place so further surfaces can
  be migrated without architectural churn.  Pseudo-locale gives QA a
  cheap layout-truncation probe.
- **Negative**: Three new crates (`fluent-bundle`, `unic-langid`,
  `intl-memoizer`).  Combined cold-compile cost on Windows host:
  ~1 minute (one-time).  Runtime cost per lookup is microseconds.
- **Neutral**: The pseudo-locale (`x-pseudo`) is intentionally
  accepted by the config validator.  A future stricter mode (release
  flag) can reject it for end-user installs.
