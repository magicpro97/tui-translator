## Architecture readiness

The production phase can add OEM/DriverSink support without modifying STT/MT/TTS orchestration:

1. `src/pipeline/audio_sink.rs` owns the `AudioSink` trait and `MockAudioSink`; sink implementations receive already-synthesized audio bytes.
2. `src/config/mod.rs` owns `TtsRouting` (`Speakers`, `VirtualMic`, `Both`) and `virtual_mic_device`; routing choices are config data, not STT/MT pipeline data.
3. `src/pipeline/playback.rs` owns `PlaybackSinkTarget`, `PlaybackRoutePlan`, and multi-sink fan-out. Adding an OEM/DriverSink path is localized to route planning and sink construction.
4. `src/pipeline/mod.rs` still performs STT -> MT -> TTS orchestration once per utterance. It does not know whether the translated audio is played through speakers, a third-party virtual cable, both, or a future production driver sink.
5. `tests/vmic_mvp_readiness.rs` and `scripts/check-vmic-mvp-evidence.ps1` enforce these seams as part of the checkpoint.
6. `.github/workflows/ci.yml` adds a VMIC-A8 readiness job for evidence aggregation, release build, and device-list smoke.
